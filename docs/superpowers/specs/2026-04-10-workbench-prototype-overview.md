# 原型总览文档：前端界面设计、功能点分布与业务链路流程

## 1. 文档目的

本文档用于对当前“普惠资产引擎”原型进行整体说明，便于后续开展以下工作：

- 产品评审与方案沟通
- 前端重构与设计统一
- 测试用例编写
- PR 说明与版本交接

本文档聚焦三个维度：

1. 前端界面设计现状
2. 功能点分布
3. 业务链路流程

---

## 2. 产品定位

当前原型定位为一套面向银行普惠金融场景的 **业务作业台**，核心场景聚焦在：

`脱核链贷`

这套原型不强调依赖链主 ERP 或链主直接确权，而是围绕以下思路展开：

- 通过内部数据和替代性证据识别链上客户
- 将客户纳入预授信和补审流程
- 对已授信客户进行风险监控与贷后经营
- 通过合作方与数据来源管理，支撑场景复制与交付

当前原型已经从“单案例演示页”逐步演进为“多样本业务作业台”，并围绕 5 个脱核链贷样本组织业务流程。

## 3. 前端界面设计现状

### 3.1 整体设计方向

当前界面风格已经从传统后台报表页，逐步收敛到：

`可信业务骨架 + 轻科技交互层`

其主要特征包括：

- 深浅结合的业务后台结构
- 模块化一级导航与二级菜单
- 样本切换器、状态胶囊、AI 洞察条等增强组件
- 大量基于卡片的工作区组织方式
- 强调“当前样本 + 当前动作 + 下一步建议”

### 3.2 当前首页风格状态

首页目前仍处于“全局控制台”与“今日工作台”之间的过渡状态。

根据最新设计方向，首页后续更建议收敛为：

`个人工作首页 / 今日业务首页`

重点强调：

- 今日优先任务
- 当前重点客户
- AI 建议
- 业务入口分发

而不是继续承担重型经营驾驶舱的职责。

### 3.3 组件与设计系统现状

当前界面设计已经形成一定的通用组件基础，主要体现在：

- 统一的页面头部与场景骨架
- 状态标签 `StatusPill`
- 通用工作面板 `WorkbenchPanel`
- 指标卡 `MetricCard`
- 样本切换器 `SampleSwitcher`
- 实体摘要卡 `EntitySummaryCard`
- 流程条、洞察条、任务卡等作业台组件

整体趋势是从“单页堆信息”向“统一作业台语法”演进。

### 3.4 当前设计上的主要特点

#### 优点

- 一级场景结构清楚
- 二级菜单定位明确
- 多数页面已摆脱单纯字段罗列
- 多样本演示已开始驱动主工作区
- AI 感更多来自“建议与判断”而非纯装饰

#### 仍需继续优化的方向

- 首页需进一步从“控制台”转向“今日工作台”
- 个别页面仍保留单案例叙事残影
- 组合层与当前样本层还需要表达得更清楚
- 个别页面仍存在“规则定义页”和“运营结果页”混杂的问题

## 5. 业务样本体系

当前原型已引入 5 个脱核链贷示例样本，统一组织在：

- [data.ts](/Users/admin/Documents/test007/硅基工作台/src/demo/chainLoan/data.ts)

这些样本共同构成“多样本业务作业台”的基础。

### 样本作用分层

当前样本体系主要用于覆盖不同经营状态：

- 已识别样本
- 预授信样本
- 补审样本
- 已授信样本
- 风险/恢复样本

样本体系的目的不是覆盖所有普惠贷款场景，而是用同一条脱核链贷主线，展示不同业务阶段下的样本状态。

### 当前样本驱动机制

系统通过 `DemoContext` 管理：

- 当前选中样本
- 当前样本阶段
- 当前样本的风险状态
- 当前样本的恢复状态

## 6. 业务链路流程

对应关系定义见：

- [data.ts](/Users/admin/Documents/test007/硅基工作台/src/demo/chainLoan/data.ts)

### 6.1 业务主链

建议统一理解为：

`生态接入 -> 客群识别 -> 预授信 -> 补审 -> 审批通过 -> 风险监控 -> 贷后恢复`

### 6.2 各阶段业务意义

#### 1. 生态接入

作用：

- 建立链主、桥接节点、借款主体、物流方、内部数据方之间的关系框架
- 明确该场景为什么可以做脱核链贷

对应页面：

- 合作方管理 / 接入总览

#### 2. 客群识别

作用：

- 基于内部流水、税票、物流、订单等信息识别链上客群候选
- 判断是否具备真实经营背景

对应页面：

- 客群识别 / 关系候选池
- 客群识别 / 关系图谱

#### 3. 预授信

作用：

- 将满足基本条件的客户纳入授信资产池
- 形成待进一步判断的预授信候选

对应页面：

- 授信资产池 / 预授信池

#### 4. 人工补审

作用：

- 在缺少链主直接确权的情况下，依赖订单、物流、回款等替代性证据进行增强审批
- 给出建议通过、降额、补材料、退回等结论

对应页面：

- 产品与审批 / 产品匹配
- 产品与审批 / 补审作业

#### 5. 审批通过

作用：

- 客户获得授信
- 进入在营资产阶段

对应页面：

- 授信资产池 / 在营资产
- 产品与审批 / 补审作业

#### 6. 风险预警

作用：

- 对已在营客户进行回款周期、履约、账户流向等风险信号监测
- 触发额度收缩、复核、观察等动作

对应页面：

- 风险监控 / 预警总览
- 风险监控 / 监控指标
- 风险监控 / 处置动作

#### 7. 贷后恢复

作用：

- 针对风险客户开展恢复观察与后续经营
- 判断是否恢复额度、继续观察、或回归正常经营

对应页面：

- 贷后经营 / 贷后总览
- 贷后经营 / 客户分层
- 贷后经营 / 增收动作

## 8. 当前版本的主要特点总结

### 8.1 已经形成的能力

1. 从单案例演示升级为多样本作业台
2. 从字段罗列升级为模块化业务工作区
3. 从单一授信演示扩展到完整前中后链路
4. 已有一定设计系统雏形和统一组件语法

### 8.2 当前仍需继续优化的方向

1. 首页继续从“控制台”收敛到“今日工作首页”
2. 组合经营层与当前样本层进一步拆清
3. 个别页面的业务口径继续收紧
4. 真实企业名与演示关系之间的口径风险需要明确提示
5. 个别页面继续收掉“案例讲述页”残影，强化“业务作业页”定位

## 9. 一句话总结

当前这套原型已经形成了一套围绕脱核链贷展开的业务作业台雏形：

- 前端界面上，已经具备统一场景结构、多样本驱动和 AI 辅助表达
- 功能分布上，覆盖了识别、入池、审批、风险、贷后、接入等主要业务域
- 业务链路上，已经搭起了从生态接入到贷后恢复的完整主线

后续的重点，不再是“补更多页面”，而是继续：

- 收紧业务口径
- 统一首页风格
- 强化当前样本驱动能力
- 提升页面之间的业务衔接一致性
