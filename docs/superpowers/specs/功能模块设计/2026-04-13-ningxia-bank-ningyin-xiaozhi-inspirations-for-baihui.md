# 宁夏银行“宁银小智”案例对百慧产品设计的借鉴清单

## 文档目的

基于宁夏银行“宁银小智”大模型在信贷风险管理中的数智化应用建设项目案例内容，提炼对 `百慧` 产品设计、PoC 规划与能力架构的可借鉴点。

本文档聚焦：

- 哪些经验可以直接吸收
- 哪些内容需要本地化调整
- 哪些点不建议照搬
- 对百慧当前产品设计与 PoC 路线的直接启发

参考案例来源：

- [宁夏银行：“宁银小智”大模型在信贷风险管理中的数智化应用建设项目](https://www.fintechinchina.com/cases/8398)

---

## 1. 对百慧最有价值的总体启发

这篇案例对百慧最大的启发，不在于“又做了一个大模型助手”，而在于它把大模型真正嵌入到了信贷业务系统中，并形成了以下几个关键特征：

1. 大模型以“系统内高频入口”的方式融入，而不是外挂工具
2. 对话式入口与结构化业务组件并存
3. 画像、深度分析、风险扫描、行业洞察等能力统一由一个智能助手承接
4. 技术上采用“大模型 + RAG + Agent + 垂直知识库”的组合方式
5. 强调“效果导向”与“持续运营”，避免伪智能化

这些点对百慧当前从“原型演示器”向“AI 作业中台”演进非常有参考价值。

---

## 2. 可直接吸收的借鉴点

### 2.1 智能能力必须深嵌系统，而不是悬浮在系统之外

案例里最值得借鉴的一点是：

`宁银小智` 不是独立门户，而是已经无缝集成在信贷管理系统内，用户可在系统主页直接唤醒。

对百慧的启发：

- 百慧不应该只是“看起来像一套 AI 产品”
- 它要在界面上明确呈现：
  - 当前样本上下文
  - 当前业务阶段
  - 当前 AI 判断
  - 当前可推进动作

对百慧的直接建议：

- 继续强化“当前样本 + 当前判断 + 下一步动作”的设计语言
- 不要把 AI 能力做成零散的展示卡片
- 要把 AI 建议作为作业流内的正式组件

---

### 2.2 对话式入口值得吸收，但不能替代结构化作业流

案例中“宁银小智”提供：

- 通用问答
- 关联问题智能联想
- 一问即答

这说明对话式入口在金融 AI 产品里有很强的接受度。

对百慧的启发：

- 百慧后续可以保留全局搜索 / 命令入口
- 但不应把结构化的识别、尽调、审批页面都替换成聊天页

对百慧的直接建议：

- 顶部命令栏可以逐步升级为：
  - 搜索客户
  - 搜索任务
  - 搜索规则
  - 问询行业 / 风险 / 尽调问题
- 同时保留结构化页面作为主工作区

---

### 2.3 企业画像、专项分析、风险归因这类“智能组件”思路很值得借鉴

案例中强调：

- 企业全景画像
- 风险归因
- 财务分析
- 科创分析
- 行业风险

它不是只给一个总答案，而是把智能能力拆成若干专项组件。

对百慧的启发：

- 百慧可以把 AI 能力拆成更清楚的组件：
  - 关系识别
  - 公私联动验证
  - 风险归因
  - 证据覆盖判断
  - 报告摘要
  - 审批建议解释

对百慧的直接建议：

- 在页面设计上继续强化：
  - `AI 判断`
  - `推荐理由`
  - `风险提示`
  - `下一步建议`
 这类组件化表达

---

### 2.4 批量风险扫描与分值排名，对 PoC 很有启发

案例里提到：

- 企业名单批量风险扫描
- 分值排名
- 风险动态变化

这对百慧当前的名单接入和识别链路非常有帮助。

对百慧的启发：

- 名单接入后，不应只停在“导入成功”
- 应该尽快输出：
  - 推荐优先级
  - 风险初判
  - 分层结果
  - 下一步动作

对百慧的直接建议：

- 继续坚持当前 PoC 路线：
  - 名单接入
  - 样本生成
  - 推荐流
  - 条件筛选
  - 候选列表
- 在这个链路里加入排序、优先级和风险初判表达

---

### 2.5 市场洞察 / 行业洞察能力值得保留为扩展方向

案例中把行业研究和产业集群数据纳入知识库，让用户能做行业和区域产业分析。

对百慧的启发：

- 百慧不必在 PoC 阶段把行业洞察做得很重
- 但应在架构和导航上给未来扩展保留空间

对百慧的直接建议：

- `智能识别` 或 `智能监控` 里后续可以加入：
  - 行业洞察卡
  - 产业链分析
  - 区域产业图谱

---

### 2.6 “重运营、轻伪智能”这条经验非常适合直接采纳

案例明确强调：

- 避免重展示轻实用
- 避免重采购轻运营
- 持续接受用户反馈、调优功能

这对百慧的启发非常直接。

对百慧的直接建议：

- PoC 阶段不要先堆花哨 AI 功能
- 先把：
  - 名单接入
  - 样本生成
  - 推荐
  - 尽调
  - 审批摘要
  这条链跑通
- 后面再逐步增强 AI 解释、行业问答、深度分析

---

## 3. 可借鉴但需要本地化调整的点

### 3.1 “一个智能助手统管全流程”值得借鉴，但需要谨慎处理产品结构

案例中 `宁银小智` 的能力覆盖：

- 问答
- 画像
- 深度分析
- 报告
- 风险扫描
- 行业洞察

对百慧的启发：

- 你可以让百慧具备统一智能助手心智
- 但在产品结构上，仍建议保持“智能识别 / 智能尽调 / 智能审批 / 智能监控 / 智能经营”的分层菜单

百慧的调整建议：

- 不要把所有能力都做成一个聊天窗口
- 要保持：
  - 统一 AI 品牌
  - 分阶段作业入口

---

### 3.2 技术架构中的 RAG、Agent、Memory 管理值得吸收，但 PoC 不宜一次铺开

案例中技术方案已经进入：

- 知识库管理
- Agent 管理
- Memory 管理
- Action 管理

对百慧的启发：

- 百慧后续完全可以朝这条路演进
- 但当前 PoC 仍应聚焦业务闭环而不是大而全平台能力

百慧的调整建议：

- PoC 阶段只需在概念上保留：
  - 规则与样本生成
  - 基础知识库
  - 报告模板
  - 后续 Agent 编排扩展点

---

### 3.3 “近千用户、全场景推广”的表述可参考，但不宜提前作为当前叙事重点

案例已全面推广到所有信贷场景。

对百慧的启发：

- 说明智能作业层模式具备扩展潜力
- 但百慧当前还处于 PoC / 原型升级阶段

百慧的调整建议：

- 对外叙事先讲：
  - 小切口试点
  - 可复制能力
  - 逐步向信贷全流程扩展
- 不要过早把自己包装成已覆盖全流程成熟平台

---

## 4. 不建议照搬的点

### 4.1 不建议把产品心智收缩成“对话助手”

虽然案例里“智能助手”心智很强，但百慧当前更适合定位为：

`AI 作业中台`

而不是：

`一个聊天机器人`

因此不建议：

- 用问答入口替代结构化工作区
- 让产品主界面过度聊天化

---

### 4.2 不建议 PoC 阶段直接采用重技术堆栈叙事

案例中提到：

- DeepSeek
- 垂直大模型
- Spring Cloud
- RAG
- Agent
- Memory
- Flask

这些内容对技术方案有意义，但不适合作为百慧当前产品设计重点。

百慧当前更应该坚持：

- 业务先行
- 页面先行
- 数据流先跑通
- 架构演进可预留

---

### 4.3 不建议把“行业洞察、营销画像、风险扫描、尽调、审批”一次都做深

案例里覆盖面很广。

对百慧来说，如果一次性都做深，会稀释 PoC 重点。

因此不建议：

- 同时重做所有一级菜单
- 同时追求全链路“看起来都很智能”

---

## 5. 对百慧当前产品设计的直接启发

### 5.1 今日工作台

借鉴点：

- 应该更像智能助手驱动的工作首页，而不是传统驾驶舱

直接动作：

- 强化：
  - 当前重点客户
  - 今日待办
  - AI 建议
  - 可快速进入下一步作业

---

### 5.2 智能识别

借鉴点：

- 名单接入后应快速形成：
  - 推荐
  - 风险初判
  - 排名
  - 下一步动作

直接动作：

- 继续按当前路线完善：
  - 文件导入
  - API 接入
  - 智能推荐流
  - 条件筛选流
  - 候选资产列表
  - 关系图谱

---

### 5.3 智能尽调

借鉴点：

- 尽调不只是“出报告”，而是：
  - 问答
  - 画像
  - 分析
  - 报告
  - 解释

直接动作：

- 强化当前设计中的：
  - 材料采集
  - 证据核验
  - 尽调报告
  - 报告中心
- 后续可增加：
  - 智能问答入口
  - 历史追问
  - 关联问题推荐

---

### 5.4 智能监控

借鉴点：

- 批量风险扫描和风险排名非常适合做成智能监控的核心亮点

直接动作：

- 智能监控页后续应强化：
  - 风险扫描
  - 风险排序
  - 风险归因
  - 风险动态变化

---

### 5.5 策略与配置

借鉴点：

- 如果未来走向更成熟平台，确实需要：
  - Agent 管理
  - 知识库管理
  - 模板管理
  - 数据源管理

直接动作：

- 当前阶段先收敛在：
  - 标准规则
  - 场景规则
  - 产品配置
  - 审批规则
  - 数据源管理
  - 报告模板

---

## 6. 对百慧 PoC 路线的直接建议

### 第一阶段：业务闭环优先

优先验证：

- 名单接入
- 样本生成
- 智能推荐
- 条件筛选
- 智能尽调
- 智能审批

### 第二阶段：强化助手能力

在主链跑通后增强：

- 智能问答
- 历史记录
- 关联问题推荐
- 行业洞察与风险归因

### 第三阶段：平台化扩展

再逐步增强：

- RAG 知识库
- Agent 编排
- Memory 管理
- 多系统联动

---

## 7. 一句话结论

这篇“宁银小智”案例对百慧最重要的借鉴，不是“也做一个大模型助手”，而是以下三条产品原则：

1. 智能能力必须深嵌业务系统，而不是浮在系统之外
2. 对话式入口可以增强体验，但不能替代结构化作业流
3. 必须坚持“效果导向”和持续运营，避免做成重展示、轻落地的伪智能产品

这三条原则非常适合作为百慧下一阶段产品演进和 PoC 验证的参考框架。
