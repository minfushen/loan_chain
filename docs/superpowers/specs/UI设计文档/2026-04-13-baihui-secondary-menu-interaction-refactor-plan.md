# 百慧二级功能菜单交互改造方案

## 文档目的

基于当前 `文件导入`、`API 接入`、`智能推荐流` 三个二级功能菜单的前端 UI/UX 设计与实现方式，提炼其背后的共性交互逻辑，并进一步输出一套适用于百慧全产品二级菜单的交互改造方案。

本文档聚焦：

- 这 3 个页面当前 UI/UX 设计的底层逻辑是什么
- 为什么它们已经比传统卡片平铺页更接近“工作台”
- 这套逻辑如何向全产品其他二级菜单迁移
- 后续二级菜单页面应遵循什么统一交互模式

本文档适用于：

- 智能识别
- 智能尽调
- 智能审批
- 智能监控
- 智能经营
- 授信资产池
- 今日工作台中的二级工作页面

## 二、这 3 个页面已经体现出的底层交互逻辑

虽然这 3 个页面在业务内容上完全不同，但从 UI/UX 角度看，它们已经共享了一套很清晰的底层逻辑。

## 1. 页面不再是“信息页”，而是“状态化工作页”

这 3 个页面的共同点，是都不再把页面理解为单纯的信息展示面，而是理解成：

**围绕某个对象、某种状态、某个下一步动作展开的工作页。**

表现为：

- 文件导入围绕“批次状态”组织页面
- API 接入围绕“接口状态 + 批次状态”组织页面
- 智能推荐流围绕“推荐对象状态 + 建议动作”组织页面

这意味着页面的核心单位不再是卡片，而是：

- `状态`
- `对象`
- `动作`

---

## 2. 页面结构从“平铺”转向“主从”

这 3 个页面已经不再是完全平铺式布局，而是明显在向主从结构靠近：

- 列表 / 批次 / 推荐对象承担左侧“主列表”
- 详情区承担中部“当前对象解释区”
- 决策区承担右侧“AI 判断与动作区”

这套结构的价值在于：

- 用户不会在多个区域来回找重点
- 页面天然形成“选对象 -> 看详情 -> 做动作”的路径

这正是工作台式交互的基础。

---

## 3. 页面开始具备“先结论，后依据，再动作”的决策语法

尤其是在 `DecisionPanel` 的使用上，这 3 个页面已经体现出一种很有价值的统一语法：

1. 先给当前判断
2. 再给关键依据
3. 再给风险或限制
4. 最后给下一步建议和动作

这与传统后台“先罗列数据，用户自己判断”完全不同。

这意味着百慧的二级菜单正在从：

**信息系统**

走向：

**智能决策工作台**

---

## 4. 页面开始具备“状态切片”和“渐进展开”的节奏

这 3 个页面共同具备两个节奏特征：

### 状态切片

- 文件导入按批次状态分组
- API 接入按接口状态、批次状态切片
- 智能推荐流按推荐等级分组

### 渐进展开

- 先看顶部 KPI
- 再看对象列表
- 再看详情
- 再看抽屉 / 页签 / 深度信息

这说明它们已经不再是一屏把所有内容扔给用户，而是在尝试形成“从全局到局部”的浏览节奏。

---

## 5. 页面开始从“可看”转向“可推进”

这 3 个页面最关键的共同点，不是长得像，而是都开始具备明确的推进动作：

- 文件导入：`生成样本`、`进入识别`
- API 接入：`重新处理`、`进入识别`
- 智能推荐流：`进入筛选`、`进入尽调`

也就是说：

页面里的信息和动作开始形成闭环。

这正是二级菜单应该具备的产品价值：

**不是用户看完信息就结束，而是看完之后能继续往下走。**

---

## 三、从这 3 个页面抽象出的交互底层模型

综合来看，这 3 个页面背后共享的不是某种单一组件，而是一套统一的交互模型。

可以概括为：

## `状态驱动 + 主从结构 + AI 决策 + 固定动作`

具体分为 4 层。

### 1. 状态驱动层

页面的组织方式必须围绕状态，而不是围绕散乱信息块。

状态可以是：

- 批次状态
- 接口状态
- 推荐等级
- 样本阶段
- 风险状态
- 审批状态

页面首先应让用户理解：

**当前对象处于什么状态。**

---

### 2. 主从结构层

页面必须建立：

- 对象选择区
- 当前详情区
- 辅助判断区

也就是：

- 左侧选对象
- 中部看详情
- 右侧做判断和动作

这是一切工作台体验的核心结构。

---

### 3. AI 决策层

页面不应只呈现原始数据，还必须统一给出：

- 当前判断
- 关键依据
- 风险提示
- 下一步建议

即便当前还是规则生成，也应以“AI 判断”的形式统一表达。

---

### 4. 固定动作层

页面必须保留 1 个主动作和 1 到 2 个次动作，且位置稳定。

不能把：

- `进入尽调`
- `进入预审`
- `生成样本`
- `人工确认`

分散在多个卡片角落。

---

## 四、这套底层逻辑对全产品的启发

这 3 个页面其实已经告诉我们：

**百慧的二级菜单不应该再按“信息块堆砌页”设计，而应该统一改造成“状态化工作页”。**

这意味着全产品二级菜单都应逐步统一到下面这条公式：

> 页面定位 = 当前对象 + 当前状态 + 当前判断 + 下一步动作

而不是：

> 页面定位 = 一组卡片 + 一组表格 + 一组图表

---

## 五、百慧二级菜单的统一交互改造方向

## 1. 二级菜单统一改造成“工作流页面”

每个二级菜单页面都应回答 4 个问题：

1. 当前处理对象是谁
2. 当前处于什么状态
3. 当前系统判断是什么
4. 当前下一步动作是什么

如果回答不了这 4 个问题，这个页面大概率还是展示页，而不是工作页。

---

## 2. 二级菜单统一采用“三段式骨架”

建议所有核心二级菜单逐步统一采用以下骨架：

### 顶部

- 面包屑
- 页面标题
- 页面副标题
- KPI Summary Bar

### 中部

- 左：对象列表 / 队列 / 状态泳道
- 中：当前对象详情 / 页签 / 依据
- 右：AI 判断与动作

### 底部

- Sticky Action Bar
- 历史记录 / 流转记录 / 深度信息

这套骨架已经在 `文件导入`、`API 接入`、`智能推荐流` 里开始出现，应当继续扩展到全产品。

---

## 3. 二级菜单统一建立“状态分组”逻辑

当前很多页面最大的问题，是对象虽然很多，但没有形成状态分层。

建议后续所有二级菜单都优先按状态组织内容，例如：

- 材料解析：待解析 / 待确认 / 已可核验
- 证据核验：待核验 / 有异常 / 已通过
- 尽调报告：待生成 / 待审阅 / 已提交 / 已归档
- 补审作业：待补件 / 待复核 / 已通过 / 已退回
- 风险探针：高风险 / 观察中 / 正常

状态分组的作用是：

- 提升扫读效率
- 降低认知负担
- 自然形成优先级

---

## 4. 二级菜单统一建立“当前选中对象”机制

这 3 个页面的一个重要优点是：

- 页面中始终有一个“当前对象”处于选中状态

这件事应该推广到所有二级菜单。

因为一旦没有“当前对象”，页面就会重新退化成：

- 看板
- 报表
- 文档页

有了“当前对象”，页面才真正开始进入工作状态。

---

## 5. 二级菜单统一引入 `DecisionPanel`

`DecisionPanel` 不应只服务于这 3 个页面，而应成为全产品核心二级菜单的标准组件。

建议统一结构：

1. 当前判断
2. 关键依据
3. 风险提示
4. 下一步建议
5. 主动作

适用页面包括：

- 材料解析
- 证据核验
- 尽调报告
- 产品匹配
- 补审作业
- 风险探针
- 客户恢复

---

## 6. 二级菜单统一引入 `StickyActionBar`

这 3 个页面已经验证了一件事：

关键动作如果固定在页面底部，会极大提升“工作台感”。

建议将它作为全产品二级菜单的统一规则：

- 每个页面都要有一个主动作
- 主动作始终可见
- 次动作不超过 2 个

---

## 六、二级菜单交互改造的标准模板

推荐后续所有二级菜单都优先套用以下交互模板。

```text
1. 顶部定位层
   - 页面标题
   - 页面副标题
   - 页面级动作

2. KPI 摘要层
   - 3~5 个关键指标

3. 主工作层
   - 左：状态分组 / 对象列表
   - 中：当前对象详情 / 页签 / 依据
   - 右：DecisionPanel

4. 深度信息层
   - 抽屉 / 折叠 / 时间线 / 日志

5. 固定动作层
   - StickyActionBar
```

这就是百慧二级菜单未来统一的交互标准骨架。

## 七、按产品模块的迁移建议

## 1. 智能识别

当前已具备较强基础，应继续强化：

- 状态分组
- 当前对象选中
- DecisionPanel
- StickyActionBar

属于最成熟的一组。

---

## 2. 智能尽调

建议全面套用这套模式：

- 材料解析
- 证据核验
- 尽调报告
- 报告中心

尤其要避免重新退回“材料表单页”和“报告列表页”的老模式。

---

## 3. 智能审批

建议重点改造：

- 产品匹配
- 预审与推单
- 补审作业
- 审批摘要

这里最适合做成：

- 左侧对象 / 队列
- 中部依据 / 规则 / 结果
- 右侧审批判断 / 动作

---

## 4. 智能监控

建议重点改造：

- 风险状态分组
- 风险对象列表
- 风险归因详情
- 处置建议与动作

避免只剩大图表和风险列表。

---

## 5. 智能经营

建议重点改造：

- 客户分层
- 当前经营状态
- 增收建议
- 恢复建议

避免只剩经营看板和静态列表。

---

## 八、当前最值得统一沉淀的前端交互原语

从目前实现看，以下原语已经证明有效，建议作为全产品统一交互原语保留和扩展：

- `PageShell`
- `KpiBar`
- `Stepper`
- `DecisionPanel`
- `StickyActionBar`

它们分别对应：

- 页面骨架
- 全局摘要
- 流程阶段
- AI 判断
- 固定动作

这实际上已经构成了百慧二级菜单的前端交互语言。

---

## 九、不建议继续沿用的旧模式

如果要完成全产品交互升级，以下旧模式建议逐步淘汰。

### 1. 卡片平铺式页面

问题：

- 无主次
- 无当前对象
- 无推进动作

---

### 2. 详情只能靠跳页查看

问题：

- 上下文断裂
- 返回成本高

---

### 3. 操作按钮散落在各区块角落

问题：

- 用户不知道最重要动作在哪

---

### 4. 只有数据，没有判断

问题：

- 页面像报表，不像 AI 作业台

---

## 十、一句话结论

`文件导入`、`API 接入`、`智能推荐流` 这 3 个页面已经验证了百慧二级菜单的正确方向：

**二级菜单不应再被设计成“内容展示页”，而应统一升级为“状态驱动、主从清晰、带 AI 判断与固定动作的工作流页面”。**

这应当成为百慧全产品二级菜单后续交互改造的统一底层逻辑。
