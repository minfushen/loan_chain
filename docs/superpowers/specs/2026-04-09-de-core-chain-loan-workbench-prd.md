# 脱核链贷业务作业台 PRD

## 1. 文档信息

- 文档名称：脱核链贷业务作业台 PRD
- 文档日期：2026-04-09
- 文档状态：Draft for Review
- 适用对象：产品经理、设计、前端、测试、售前方案
- 关联原型：`硅基工作台` 当前前端原型

## 2. 产品定义

### 2.1 产品定位

本产品定位为一套面向银行普惠业务团队的脱核链贷业务作业台，用于支持银行在无法直接接入链主 ERP、无法获得链主强确权的条件下，依托银行内部数据、票据数据、回款数据、物流数据与规则引擎，识别链上小微主体、完成补审决策、实施贷中预警与贷后恢复经营。

本产品不是面向客户经理的通用工作台，也不是泛化的数据中台或营销平台，而是围绕单一业务闭环构建的专业业务作业台。

### 2.2 核心业务问题

本产品解决以下四类核心问题：

1. 银行难以稳定接入链主 ERP，传统供应链金融识别链条能力不足。
2. 银行无法仅依赖核心企业确权覆盖链上三级、四级小微供应商与末端服务主体。
3. 审批人员在脱核场景下缺乏足够强的替代性证据，不敢批、不好批。
4. 风险识别和贷后经营割裂，放款后缺少动态预警、额度联动和恢复经营闭环。

### 2.3 产品目标

本产品阶段性目标如下：

- 目标一：让业务人员识别“待授信企业为什么属于这条产业链”
- 目标二：让审批人员看懂“凭什么在没有链主强确权时仍然可以批”
- 目标三：让风险人员看到“预警如何触发、额度如何联动”
- 目标四：让贷后人员看到“风险不是终点，恢复经营如何继续推进”

## 3. 目标用户

### 3.1 主要用户角色

1. 普惠产品经理
职责：维护规则、定义场景产品、查看场景转化效率

2. 补审审批人员
职责：查看证据链、进行人工补审、决定是否授信

3. 风险经理
职责：查看风险探针、跟踪预警、执行自动或人工处置

4. 贷后经营人员
职责：管理恢复观察池、执行恢复经营、推动续贷与增收

5. 业务管理者
职责：查看作业台总体状态、关注资产生成效率与风险质量

### 3.2 非目标用户

- 外部借款客户
- 零售客户经理日常办公用户
- 通用 CRM 用户
- 核心信贷系统操作员

## 4. 业务范围

### 4.1 业务主线

产品围绕一条完整脱核链贷作业链路展开：

1. 合作方接入
2. 客群识别
3. 预授信生成
4. 人工补审
5. 审批通过
6. 风险预警
7. 恢复经营

### 4.2 MVP 范围

本期原型和需求范围聚焦以下页面与功能：

- 合作方管理
- 客群池
- 授信资产池
- 产品与审批
- 风险监控
- 贷后经营

本期不包括：

- 真正的外部系统接入
- 真正的授信审批流转
- 真正的用户权限体系
- 真正的数据检索、导出、消息通知

## 5. 业务场景定义

### 5.1 样例业务场景

- 链主企业：宁川新能源
- 桥接节点：盛拓模组科技
- 待授信企业：常州衡远包装材料有限公司
- 企业角色：新能源电池产业链三级包装材料供应商
- 目标产品：订单微贷
- 拟授信额度：120 万
- 风险事件：回款周期拉长 + 物流履约延迟
- 贷后动作：临时收缩额度至 96 万，满足恢复条件后恢复部分或全部额度

### 5.2 业务判断方式

本产品不以“链主 ERP 直连”为唯一前提，而采用以下替代性识别逻辑：

- 内部对公流水识别稳定交易对手
- 发票与结算口径识别经营连续性
- 物流单据识别履约佐证
- 回款归集识别资金闭环
- 规则引擎形成关系候选、预授信、补审与预警动作

## 7. 页面级需求

### 7.1 场景驾驶舱

目标：以业务总览方式展示当前作业台状态，而不是展示解决方案宣传内容。

核心需求：

- 展示当前案例摘要
- 展示步骤条
- 展示资产生成、预授信、补审、风险、恢复等关键概览指标
- 支持进入各业务模块

页面要求：

- 不出现方案型长段文案
- 首屏应以业务字段与状态卡为主

### 7.2 客群池

目标：明确识别逻辑与客群分层方式，让业务人员知道“为什么识别出这个客户”。

核心需求：

- 关系候选池展示候选企业、链条、角色、评分和当前动作
- 关系图谱突出链主、桥接节点和待授信企业之间的主链关系
- 公私联动模块展示内部数据联动识别逻辑
- 标准小微模块只展示规则定义，不展示经营结果为主内容
- 长尾场景模块展示脱核链贷等非标准客群的识别逻辑

页面要求：

- 链主关系必须一眼可见
- 衡远包装在各模块中的角色表述必须一致
- 标准小微与长尾场景的定位必须清晰区分

### 7.3 授信资产池

目标：把识别结果转换为授信状态流转。

核心需求：

- 转化看板展示组合级转化概览
- 预授信池展示符合准入条件的客户
- 补审队列展示需人工判断客户
- 在营资产展示已授信客户

页面要求：

- 组合数据与案例数据要有边界
- 当前案例状态必须和步骤条一致

### 7.4 产品与审批

目标：支持业务规则配置与单户人工补审。

核心需求：

- 引擎配置展示规则治理信息
- 规则清单展示规则编号、触发条件、动作链、数据来源和效果信息
- 补审作业支持三屏交叉验证
- 审批动作支持在人工补审阶段推进

页面要求：

- 补审页是本产品的核心页面
- 必须体现“没有链主强确权时，如何通过订单、物流、回款交叉验证敢批”

### 7.5 风险监控

目标：展示贷中预警、自动动作和人工处置逻辑。

核心需求：

- 预警总览展示额度收缩过程
- 监控指标展示规则触发前后差异
- 处置动作展示自动动作与人工动作
- 规则效果展示组合层面的规则贡献

页面要求：

- 风险未触发前不得提前展示已命中的风险结果
- 风险触发后各子页口径必须一致

### 7.6 贷后经营

目标：形成从预警到恢复经营的完整闭环。

核心需求：

- 恢复作业展示恢复进度、恢复条件、时间轴与恢复按钮
- 客户分层展示组合经营策略
- 增收动作展示提额、续贷、交叉销售等经营动作
- 动作模板展示贷后动作模板

页面要求：

- 在风险未发生前不得默认进入恢复态
- 恢复页必须体现“风险不是终点，而是继续经营的开始”

### 7.7 合作方管理

目标：说明案例起点与数据来源口径。

核心需求：

- 接入总览展示链主、桥接方、待授信主体与外部数据方
- 数据来源展示内部和外部可用数据，不依赖 ERP 强接入
- 银行模板和实施路径展示产品配置与落地方式

页面要求：

- 合作方管理属于作业台背景页，不应篡改单户作业状态

## 8. 核心功能需求

### 8.1 Demo 流程状态管理

需求：

- 全局维护统一案例状态
- 页面切换时状态不丢失
- 阶段推进驱动页面内容变化

关键状态：

- ecosystem
- identified
- pre_credit
- manual_review
- approved
- risk_alert
- post_loan_recovery

### 8.2 三屏交叉验证

需求：

- 订单流、物流轨迹、回款流水三栏联动
- 用户点击不同 flow 时内容同步刷新
- 页面明确展示匹配关系

### 8.3 风险触发与额度联动

需求：

- 风险触发后展示额度从 120 万到 96 万的动态变化
- 风险探针报告同步展示
- 处置动作与规则链一致

### 8.4 恢复经营

需求：

- 恢复条件可视化
- 恢复按钮可执行
- 恢复后额度、状态、时间轴同步更新

## 9. 内容一致性要求

### 9.1 角色表达一致

待授信企业、链主、桥接节点、服务方的角色定义必须在以下页面保持一致：

- 关系候选池
- 关系图谱
- 补审作业
- 风险监控
- 贷后经营

### 9.2 状态表达一致

风险是否触发、额度是否收缩、恢复是否完成，必须在以下位置保持一致：

- 步骤条
- 案例摘要卡
- 风险总览
- 监控指标
- 恢复作业

### 9.3 页面定位一致

- 标准小微页：规则定义页
- 长尾场景页：场景识别页
- 规则效果页：组合经营页
- 补审作业页：单户作业页

## 10. 非功能要求

### 10.1 原型展示要求

- 视觉定位为银行业务作业台
- 避免方案宣传页风格
- 避免过长说明文字
- 页面以字段、状态、卡片、表格为主

### 10.2 可测试性要求

- 页面口径清晰
- 状态切换可重复
- 关键按钮有明确反馈
- 可支撑手工测试与演示测试

## 11. 验收标准

### 11.1 业务验收

- 业务人员能看懂链主关系与识别逻辑
- 审批人员能看懂补审证据链
- 风险人员能看懂风险触发与动作链
- 贷后人员能看懂恢复经营逻辑

### 11.2 原型验收

- 关键流程可点击走通
- 各页面角色和口径无明显冲突
- 关键阶段状态一致
- 不再保留明显方案型大段文案主导页面

5. 引入更清晰的组合视图 / 案例视图切换标识
