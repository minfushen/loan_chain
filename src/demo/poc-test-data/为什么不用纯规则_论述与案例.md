# 为什么不用纯规则引擎——三个来自真实数据的案例

> 本文档结合衡远包装POC演示数据，用三个具体案例说明规则引擎的边界，以及决策/审核智能体在哪里创造了增量价值。

---

## 核心论点

规则引擎解决的是**已知的确定性问题**：
> "集中度超过55%，必须降额。"——这是规则，不需要推理。

智能体解决的是**已知规则无法覆盖的灰色地带**：
> "集中度72%，但买方是链主直接指定的二级供应商，合同锁定3年——这个集中度是风险还是强绑定？"——这需要推理。

---

## 案例一：公私混用异常（最有说服力）

### 数据来源
交通银行流水 `交通银行流水_衡远包装_2025Q4.csv`

### 事实
- 对公账户（382***961）与个人账户陈**（6228***2378）在2025年12月发生 **44次双向转账**
- 转入陈**：24笔，合计 **205.4万元**
- 转出陈**：20笔，合计 **127.7万元**
- 单笔最大转出：**45万元**（12月8日）

### 规则引擎怎么处理

```
IF 对公账户与个人账户互转金额 > 50万/月
THEN 触发"公私混用预警"
→ 结果：预警触发，案件标红，等待人工处理
```

**规则引擎到此为止。** 它只能告诉你"有问题"，无法告诉你"是什么问题"。

### 智能体怎么处理

**Round 1 — 建立假设空间（规则引擎做不到）**
```
可能性A：法人正常提现/分红（15%）
可能性B：财务人员资金归集账户（72%）
可能性C：资金挪用/垫资（8%）
可能性D：关联方资金周转（20%）
```

**Round 2 — 交叉验证排除（规则引擎做不到）**
- 调用三流验证数据：三流匹配率98.6%，回款路径完整 → **排除C（挪用）**
- 调用个人流水：周海峰月均分红3.5-5.5万，来源清晰 → **排除A（法人提现，因为陈**不是周海峰）**
- 分析转账模式：收款后立即转出，余额不沉淀 → **支持B（资金归集）**

**Round 3 — 知道自己的边界（规则引擎做不到）**
```
当前最高可能性B置信度72%，未达85%阈值
→ 不强行结案
→ 主动发起补充核实请求：请提供陈**与企业的关系证明
→ 48小时超时则标注待确认，不阻塞授信流程
```

### 差异总结

| | 规则引擎 | 智能体 |
|--|---------|--------|
| 能做什么 | 发现异常存在 | 判断异常性质 |
| 输出 | "有公私混用" | "72%可能是资金归集，需补充证明" |
| 对人工的价值 | 增加工作量（标红等待处理） | 减少工作量（已排除挪用，只需确认身份） |

## 案例二：Q3现金流为负（季节性 vs 经营恶化）

### 数据来源
农业银行流水 `农业银行流水_衡远包装_2025Q3Q4.csv`  
现金流量表 `三大财务报表_衡远包装_2025.json`

### 事实
- Q3（7-9月）经营活动现金流净额：**-10,809元**（负值）
- 同期Q3营业收入：**80万元**（全年最高季度之一）
- 农业银行8月21日单日税费扣缴：**67,609 + 460,000 = 527,609元**
- 农业银行9月23日电费托收：**151,326元**
- 农业银行9月25日税费扣缴：**191,421元**

### 规则引擎怎么处理

```
IF 季度经营现金流 < 0
THEN 触发"现金流恶化预警"
→ 结果：预警触发，降低授信评分
```

**规则引擎的判断是错的。** 这个负值不是经营恶化，是旺季备货导致的集中付款。

### 智能体怎么处理

**Step 1 — 识别异常的时间模式**
```
Q3现金流为负，但Q3营收是全年最高 → 矛盾信号
→ 触发"矛盾信号分析"子任务
```

**Step 2 — 定位负值来源**
```
调用工具：bank_flow_analyze(农业银行流水, 2025-Q3, 大额支出)
→ 发现：8月21日单日支出超60万（税费+电费）
→ 发现：9月23-25日集中支出超34万（电费+税费）
→ 结论：负值来源是季度末集中缴税和电费，不是经营支出恶化
```

**Step 3 — 与行业规律交叉验证**
```
调用工具：industry_benchmark(包装材料制造业, 季节性现金流)
→ 包装材料Q3为旺季，备货付款集中在季末
→ 税费在季末集中扣缴（增值税按季申报）
→ 结论：Q3现金流为负属正常季节性波动，不是风险信号
```

**最终输出**
```
Q3经营现金流净额-10,809元，原因：季末集中缴税（Q3增值税4,200元+附加税）
及电费托收（151,326元），属正常季节性波动。
Q4现金流净额+254,634元，全年经营现金流净额+404,550元，整体健康。
```

### 差异总结

| | 规则引擎 | 智能体 |
|--|---------|--------|
| 判断 | 现金流恶化，降低评分 | 季节性波动，不影响评分 |
| 准确性 | **错误** | **正确** |
| 原因 | 规则不区分"为什么负" | 能追溯负值的具体来源 |

## 案例三：客户集中度72%（风险 vs 强绑定）

### 数据来源
三流交叉验证 `三流交叉验证_衡远包装_2025.json`  
系统样本数据 `data.ts`

### 事实
- 盛拓模组科技占衡远包装全年订单的 **72.1%**
- 行业规则阈值：集中度 > 55% 触发预警，> 70% 建议降额
- 但：盛拓模组是宁川新能源（链主）的直接二级供应商
- 且：衡远包装与盛拓模组的交易连续 **14个月**，关系强度 **92分**
- 且：三流匹配率 **100%**，回款周期稳定在 **30-35天**

### 规则引擎怎么处理

```
IF 最大客户集中度 > 70%
THEN 建议降额至推荐额度的80%
→ 结果：120万 → 96万
```

**规则引擎的降额是机械的。** 它不区分"集中度高是因为客户单一"还是"集中度高是因为深度绑定在产业链里"。

### 智能体怎么处理

**Step 1 — 识别集中度的成因**
```
调用工具：supply_chain_analysis(盛拓模组, 宁川新能源)
→ 盛拓模组是链主宁川新能源的直接二级供应商
→ 衡远包装是盛拓模组的指定包装材料供应商
→ 这是产业链结构性绑定，不是单一客户依赖
```

**Step 2 — 评估集中度的稳定性**
```
调用工具：transaction_stability_check(盛拓模组, 14个月)
→ 连续14个月无断单
→ 回款周期标准差仅2.3天（极度稳定）
→ 订单金额逐季增长（Q1→Q4增长5倍）
→ 结论：这是增长型绑定，不是依赖型风险
```

**Step 3 — 与规则引擎协作（不是替代）**
```
规则引擎结论：集中度72%，建议降额
智能体补充：集中度来源于产业链结构性绑定，交易稳定性极高
→ 最终建议：维持120万，但在授信条件中加入"回款归集至本行"要求
→ 理由：通过回款归集监控，可以实时感知盛拓模组的付款状态变化
```

### 差异总结

| | 规则引擎 | 智能体 |
|--|---------|--------|
| 判断 | 集中度超阈值，降额 | 产业链绑定，维持额度+加条件 |
| 对客户的影响 | 少贷24万，可能流失客户 | 足额授信，通过条件管控风险 |
| 风险管理质量 | 粗放（一刀切） | 精准（针对性条件） |

## 面试话术建议

**开场白（30秒）：**
> "我用三个来自真实银行流水的案例来回答这个问题。这三个案例都是规则引擎会给出错误或不完整答案的场景，而智能体能给出更准确、更有价值的判断。"

**核心论点（一句话版本）：**
> "规则引擎处理的是'已知的确定性'，智能体处理的是'已知规则无法覆盖的灰色地带'。银行信贷里最难的决策恰恰都在灰色地带。"

**被追问"智能体会不会出错"时：**
> "会。所以我们的设计原则是：智能体不替代人工，而是把'需要人工判断的点'精准定位，并把相关证据结构化呈现。智能体出错的代价，远小于人工在200页流水里找不到关键信号的代价。"

**被追问"可解释性"时：**
> "每一步推理都有工具调用记录和置信度变化。案例一里，智能体最终给出的不是'通过'或'拒绝'，而是'72%可能是资金归集，需要补充陈**的身份证明'——这个结论是可以被审计的，也是银行合规要求的。"
